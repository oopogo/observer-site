# 관제 세션 정책

## 1. 세션 종류

- `chat`: observer-site 관제 채팅 세션. 사용자 질문/지시와 에이전트 답변을 빠르게 주고받는 용도.
- `work`: 에이전트의 실제 작업 세션. 파일 수정, 검증, 커밋, 장기 작업 맥락을 유지하는 용도.
- `subagent`: 작업 세션에서 분리된 하위 작업 인스턴스.
- `cron`: 정기 실행 작업 세션.
- `acp`: Codex/Claude Code 등 ACP 계열 작업 세션.

## 2. 관제 채팅 세션 전환 기준

관제 채팅은 빠른 응답이 목적이므로 모델 최대 컨텍스트 100만을 끝까지 쓰지 않는다.

- 호출형 에이전트(`CALL-ONLY`): 50,000 tokens 이상이면 새 채팅 세션 전환 후보.
- 일반 관제 채팅: 80,000 tokens 이상이면 새 채팅 세션 전환 후보.
- 모든 관제 채팅: 120,000 tokens 이상이면 강제 전환 후보.
- `running/processing/queued/pending` 상태가 60초 이상 갱신되지 않으면 stuck 채팅 세션으로 보고 새 세션 전환 후보.
- 6시간 이상 오래된 관제 채팅 세션도 전환 후보.

새 채팅 세션에는 최근 대화, 미완료/실패 작업, 주요 결정, 최근 완료보고 후보를 담은 handoff 패킷을 붙인다.

## 3. 작업 세션 전환 기준

작업 세션은 작업 맥락이 중요하므로 자동 전환을 보수적으로 한다.

전환 후보:

- 컨텍스트 사용률 90% 이상.
- 세션 손상/stuck/timeout 반복.
- 사용자가 명시적으로 새 작업 세션을 요청.

작업 세션 handoff에는 다음을 포함해야 한다.

- 작업 목표
- 현재 단계
- 완료한 것
- 수정한 파일
- 검증 결과
- git 상태/커밋 여부
- 실패/차단 사유
- 남은 TODO
- 실행 중 subagent/background process
- 관련 링크/리포트/artifact
- 금지사항/주의사항

## 4. 상태 표시 원칙

- `작업 중`: 실제 running/processing/queued 세션이 최근 갱신 중일 때만 사용.
- `응답 처리 중`: 채팅 요청을 받았고 아직 짧은 시간 내 회수 중일 때 사용.
- `세션 처리 지연` 또는 `응답 회수 확인 필요`: 세션이 끝났거나 stale인데 채팅 답변이 미반영된 경우.
- `보고 대기`: 작업 evidence는 있으나 최종 사용자 보고가 아직 없을 때.
- `점검 필요`: 실패/timeout/stuck을 관제가 확인해야 할 때.

## 5. 자동 복구 원칙

- 채팅 세션은 stuck/large/old 조건에서 자동으로 새 채팅 세션을 만들 수 있다.
- 작업 세션은 자동 전환 전 반드시 handoff를 만들고, 기존 작업 세션/프로세스는 중단하지 않는다.
- subagent는 인스턴스이므로 종료된 것은 카드에서 숨기고 보관 대상이 될 수 있다.

## 6. 검증 항목

- `/api/agents`에 `chatSessionKind`, `latestSessionKind`, `chatSessionTokens`, `chatRolloverCandidate`, `workHandoffCandidate`가 표시되는지 확인.
- 큰 채팅 세션이 다음 send에서 새 관제 채팅 세션으로 전환되는지 확인.
- 기존 작업 세션이 새 채팅 세션 전환 때문에 중단되지 않는지 확인.
- handoff 패킷이 새 채팅 세션 첫 요청에 포함되는지 확인.
